# Protected surfaces and generated code

Use this reference when establishing the cleanup boundary, or when deciding whether a specific file or symbol counts as a protected surface, generated code, or vendored territory. It expands the summaries in SKILL.md — the Scope contract there remains the canonical allow / deny list.

## Protected surfaces

Treat the following as public contracts or externally observed behavior. Do not edit them as part of cleanup unless the user explicitly approved it and verification is strong:

- public exports and re-exports
- routes, controllers, handlers, resolvers, and cli flags
- request and response schemas, serializers, dto shapes, and validation rules
- exported types, interfaces, classes, and functions
- database schemas, migrations, and queries with externally observed behavior
- serialization keys, event names, cache keys, env var names, and feature flags
- framework lifecycle hooks, decorators, annotations, reflection-based wiring, plugin registration, and dependency-injection registration
- observable log messages or error text that tests, monitors, or callers may match against
- files or paths used for runtime loading

## Style signal sources

Detect the conventions the project already uses before editing, and prefer them over generic best practices:

- formatter and linter config (`.prettierrc`, `eslint.config.*`, `ruff.toml`, `biome.json`, `.editorconfig`, etc.)
- naming conventions from 5 to 10 nearby files
- file layout patterns
- import ordering conventions
- test naming and organization

## Generated and vendored code

Never modify the following without explicit approval, even if they look dirty:

- dependency directories: `node_modules/`, `vendor/`, `target/`, `.venv/`, `venv/`, `pods/`, `bower_components/`, `jspm_packages/`
- build output: `dist/`, `build/`, `out/`, `.next/`, `.nuxt/`, `bin/`, `obj/`
- lockfiles: `package-lock.json`, `pnpm-lock.yaml`, `yarn.lock`, `bun.lockb`, `cargo.lock`, `poetry.lock`, `gemfile.lock`, `composer.lock`, `go.sum`
- generated sources: anything with headers like `code generated by ... do not edit`, `@generated`, `*.pb.go`, `*.pb.cc`, `*_pb2.py`, or generated graphql types
- generated sdk clients from openapi, graphql, grpc, or smithy schemas — often under `generated/`, `__generated__/`, `gen/`, or with naming like `*-sdk.ts`, `*Api.ts`, `*Client.kt`
- database migrations already applied in production
- test snapshots unless the user explicitly asked for snapshot cleanup
- vendored third-party code under `third_party/`, `vendor/`, or `external/`

If cleanup would require regenerating generated code, stop and ask.
